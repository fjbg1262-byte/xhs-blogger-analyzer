# XHS 博主分析助手：分发与分享方案

这份文档写给项目维护者，用来规划怎么把工具分享给普通用户。

## 当前产品形态判断

当前工具由两部分组成：

- 本地 Web 工具：负责采集任务、分析、报告页面。
- Chrome / Edge 插件：负责在小红书博主主页上一键发起分析。

现阶段还不是“普通用户双击一个安装包就能用”的成熟形态。低成本内测可以先用压缩包分发，但必须配清楚图文教程。

## 最低成本内测分发方式

### 推荐方式

先做一个“Windows 内测压缩包”：

```text
xhs-blogger-analyzer-beta.zip
```

建议使用项目内置脚本生成压缩包，不要手动右键压缩整个项目文件夹：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/package_beta.ps1 -Version beta
```

生成后的文件在：

```text
release/xhs-blogger-analyzer-beta.zip
```

这个脚本会尽量排除本地运行数据、报告、数据库、Cookie、`.env`、`node_modules` 等不应该发给用户的内容。

压缩包里保留：

```text
install_windows.bat
start_windows.bat
browser-extension/
frontend/
backend/
docs/USER_GUIDE.md
requirements.txt
package.json
frontend/package.json
```

用户拿到后只需要：

1. 解压。
2. 双击 `install_windows.bat`。
3. 双击 `start_windows.bat`。
4. 按 `docs/USER_GUIDE.md` 安装浏览器插件。

### 分享渠道

适合小范围内测：

- 飞书文件
- 微信文件
- 百度网盘
- 夸克网盘
- GitHub Release

不建议一开始直接大规模公开扩散，因为：

- 用户电脑环境差异很大。
- Python / Node.js 安装仍然会卡住一部分人。
- 插件开发者模式安装对小白仍然有门槛。
- 需要先收集真实使用反馈和失败截图。

## 没有 GitHub 账号的人怎么用

GitHub 账号不是必须的。

如果项目上线 GitHub，普通用户可以这样下载：

1. 打开项目 GitHub 页面。
2. 点击右侧或顶部的 `Releases`。
3. 找到最新版本。
4. 下载 `xhs-blogger-analyzer-beta.zip`。
5. 解压后按 `USER_GUIDE.md` 操作。

不要让普通用户点 `Code -> Download ZIP`，因为那通常是源码包，不一定包含你准备好的说明、脚本和版本文件。

## 小红书引流分发建议

发布内容不要强调“爬虫”“批量采集”“防风控”。

建议表达成：

```text
XHS 博主分析助手测试版
适合新手博主和小运营团队
打开对标账号主页，一键生成本地分析报告
```

可以强调：

- 本地使用
- Chrome / Edge 插件
- 近 100 篇公开内容分析
- 自动生成主题地图、证据样本、行动计划
- 不需要手动复制 Cookie
- 不需要配置 API Key 也能用基础报告

不要强调：

- 批量抓取
- 绕过限制
- 风控对抗
- 自动化监控
- 账号池
- 保证爆款
- 保证涨粉

## 第一阶段交付包建议

内测包应该包含：

- `xhs-blogger-analyzer-beta.zip`
- `安装使用说明.md` 或 `docs/USER_GUIDE.md`
- 1 张安装流程图
- 1 张插件使用截图
- 1 张报告结果截图
- 常见问题说明

用户最容易卡住的地方：

- 不知道什么是“解压”
- 不知道在哪里打开扩展管理页
- 不知道怎么选择 `browser-extension` 文件夹
- 关闭了黑色窗口导致插件提示本地工具没启动
- 没有登录小红书
- 打开的不是博主主页

发布时要把这些卡点提前写清楚。

## 更适合普通用户的后续形态

### 方案 A：继续压缩包分发

成本最低，适合 10-50 人内测。

缺点：

- 需要用户安装 Python / Node.js。
- 插件需要开发者模式安装。
- 电脑小白仍然容易卡住。

### 方案 B：GitHub Release

适合公开透明、开源分发。

优点：

- 版本清晰。
- 用户不需要 GitHub 账号也能下载。
- 方便更新说明和问题反馈。

缺点：

- 国内用户访问 GitHub 可能慢。
- 对小白仍然不够友好。

### 方案 C：Windows 一键安装包

更适合普通用户。

目标形态：

```text
下载安装包 -> 双击安装 -> 桌面出现图标 -> 打开浏览器插件 -> 使用
```

这需要后续做：

- 后端打包为本地可执行程序。
- 前端构建为静态文件并由后端托管。
- 自动启动本地服务。
- 提供托盘图标或桌面快捷方式。
- 插件仍需单独安装，或后续上架浏览器商店。

### 方案 D：浏览器商店上架

最适合长期分发。

可选渠道：

- Chrome Web Store
- Microsoft Edge Add-ons

优点：

- 用户点击安装即可。
- 不需要开发者模式。
- 更新更方便。

缺点：

- 需要审核。
- 权限说明要非常克制。
- Cookie 权限、站点权限、隐私说明必须写清楚。
- 仍然需要本地 Web 工具配合。

## 推荐路线

### 第 1 步：小范围内测

用压缩包 + 小白说明。

目标：

- 验证用户能不能装上。
- 记录最常见失败点。
- 看报告是否真的被用户使用。

人数建议：5-20 人。

### 第 2 步：GitHub Release

当安装说明和脚本稳定后，把每个版本打成 Release。

目标：

- 方便下载。
- 方便版本回滚。
- 方便收集 issue。

### 第 3 步：做一键启动包

减少 Python / Node.js 安装门槛。

目标：

- 用户不需要打开命令行。
- 用户只需要双击启动。
- 本地服务启动失败时有明确提示。

### 第 4 步：插件上架商店

当产品稳定、权限说明和隐私说明完善后，再考虑上架。

## 分发时的风险边界

对外说明建议写清楚：

- 本工具只用于分析公开主页内容。
- 本工具不提供批量监控能力。
- 本工具不承诺涨粉或爆款。
- 用户需要使用自己的浏览器登录状态。
- 分析结果是基于样本的观察，不是平台官方数据。
- 不建议高频、批量、连续分析大量账号。

## 当前下一步建议

现在最应该做：

1. 先运行 `scripts/package_beta.ps1` 生成一份干净内测包。
2. 自己把 `release/xhs-blogger-analyzer-beta.zip` 解压到一个新文件夹，完整走一遍 `USER_GUIDE.md`。
3. 找 3-5 个完全不懂技术的人，按 `USER_GUIDE.md` 安装。
4. 让他们只截图，不解释，观察卡在哪里。
5. 把卡点改成更傻瓜的脚本或图文。
6. 再决定是否做 GitHub Release。

真正决定能不能传播的，不是功能有多强，而是：

```text
用户能不能在 10 分钟内装上
用户第一次点击后能不能看到报告
用户看完报告后有没有一个立刻能做的动作
```
