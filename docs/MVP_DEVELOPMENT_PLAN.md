# 小红书分析助手开发计划：交付结果层重构

## 1. 当前状态

已经完成：

- 本地 FastAPI 后端。
- SQLite 本地存储。
- Chrome / Edge 插件。
- 插件一键分析当前博主。
- 本地任务创建和自动打开报告页。
- Spider_XHS 采集链路。
- 基础规则分析。
- 9 维 Markdown 报告生成。

当前主要问题：

- 报告是模板驱动，读起来生硬。
- AI 默认未介入，且 LLM 配置为空。
- 现有 9 维报告有价值，但没有产品化组织。
- 数据质量没有显式评分。
- 分类硬编码，导致“其他/未分类”比例过高。
- 存在基础数据 bug：负月份、分位数反常。
- 存在反向推荐：低表现标题公式被推荐。
- 标签、正文、top/bottom 对比等字段缺失时仍可能生成强结论。
- 当前质量分主要检查文件、字段和重复证据，不能代表语义质量。
- 结论过度依赖均值，缺少中位数和单篇极端值贡献检查。
- 证据池偏向高赞样本，缺少普通样本、低表现样本和反例。
- 固定文案可能与数字矛盾，例如“41 篇但产量低”“双方均为 0 但一方更高”。
- 正文和标签为局部补采样本时，部分报告仍可能按全量样本表述。

下一阶段目标：

> 建立一套稳定的交付结果生成链路：数据校验 -> 非 AI 专家分析 -> 结构化洞察 -> 正反证据绑定 -> 产品化报告 -> 语义质量检查 -> 可选 AI 解读。

## 2. 新目标架构

新的分析链路：

```text
采集数据
  ↓
data_quality.json
  ↓
results.json
  ↓
insight_brief.json
  ↓
report_quality_check.json
  ↓
前端产品化报告页
  ↓
可选 AI 解读版
```

文件职责：

- `all_notes.json`：采集到的原始笔记。
- `contents.json`：正文或内容片段。
- `profile.json`：账号基础信息。
- `results.json`：量化指标计算结果。
- `data_quality.json`：数据完整度和可信度。
- `insight_brief.json`：结构化洞察与证据。
- `report_quality_check.json`：报告质量检查结果。
- Markdown 报告：保留为导出物，不再作为主体验。

补充说明：

- `profile.json` 需要逐步补齐 `avatar_url`、`bio`、`follower_count` 等字段，用于报告首页账号卡片。
- `contents.json` 只能保存真实正文和标签，不能用点赞数等占位字段伪装正文。
- 图片/视频媒体数据暂不进入默认主链路，后续作为可选深度分析。

## 3. 开发原则

1. 先修事实层，再做表达层。
2. 强结论必须有证据。
3. 数据不足就降级，不硬写。
4. 9 维分析不删除，改成模块化呈现。
5. AI 做表达和归纳，不做事实来源。
6. 每次生成都要可复查、可追溯、可回归测试。
7. 先区分事实、观察和假设，再生成自然语言。
8. 没有充分证据时交付验证方案，不交付确定性策略。

## 4. 阶段一：数据可信度与基础 bug 修复

### 4.1 目标

让底层数据和基础指标可信，避免报告出现明显错误。

### 4.2 修改范围

重点文件：

- `analyze_all.py`
- `backend/services/analyzer.py`
- `generate_reports.py`
- `backend/tasks/worker.py`

### 4.3 具体任务

1. 修复分位数计算。
   - 当前存在 `p25 > p75` 的问题。
   - 应确保 p25 <= median <= p75。

2. 修复时间跨度。
   - 当前存在 `content_span_months = -11`。
   - 应使用最早日期到最晚日期计算正数月份。

3. 统一互动字段解析。
   - 点赞数字符串转整数。
   - 缺失字段计入数据质量，不默认当作 0 强分析。

4. 识别正文完整度。
   - 统计 `content_text` 是否存在。
   - 统计正文长度分布。

5. 识别标签完整度。
   - 统计标签字段是否存在。
   - 标签为空时输出数据不足。

6. 识别评论/收藏字段完整度。
   - 如果字段不存在，不生成相关强结论。

### 4.4 新增输出

新增：


结构：

```json
{
  "sample_count": 100,
  "title_completeness": 1.0,
  "like_completeness": 0.87,
  "time_completeness": 1.0,
  "content_text_completeness": 0.42,
  "tag_completeness": 0,
  "comment_completeness": 0,
  "favorite_completeness": 0,
  "overall_grade": "medium",
  "warnings": [
    "标签数据不足，SEO 模块将降级",
    "评论数据不足，用户痛点分析将降级"
  ]
}
```

### 4.5 验收标准

- 不再出现负月份。
- 不再出现 p25/p75 反常。
- 数据质量文件能正确生成。
- 标签为空时不输出强 SEO 建议。
- 正文缺失时不输出强正文结构建议。

## 5. 阶段二：重构主题分类与模式识别

### 5.1 目标

降低“其他/未分类”比例，让内容分类更贴近真实账号。

### 5.2 当前问题

现有规则分类过硬，导致真实账号中 77% 内容进入“其他/未分类”。这会让后续策略失真。

### 5.3 新策略

采用“宽特征 + 聚类/归纳 + 统计验证”的方式。

处理流程：

```text
标题/正文片段
  ↓
关键词、实体、场景、对象提取
  ↓
初步聚类
  ↓
主题命名
  ↓
每类表现统计
  ↓
低置信主题标记
```

### 5.4 第一版实现方式

先不引入复杂 embedding，采用轻量方案：

1. 从标题和正文片段提取高频词、时间词、对象词、场景词。
2. 用可配置关键词规则做宽分类。
3. 对无法分类的内容进行高频词分组。
4. 生成 5-8 个动态主题。
5. 主题命名优先使用实际高频对象，而不是固定分类。

例如：

- 老手机摄影实验。
- 北京城市出行。
- AI 摄影工具观察。
- 自然风景拍摄。
- 个人体验记录。

### 5.5 后续增强

如果配置了 LLM：

- 用 LLM 对 100 条标题和正文摘要做主题归纳。
- 但必须返回结构化 JSON。
- 每个主题必须包含 note_ids。
- 后端再统计每类表现。

### 5.6 新增输出

在 `insight_brief.json` 中加入：

```json
{
  "topic_clusters": [
    {
      "id": "topic_1",
      "name": "老手机摄影实验",
      "confidence": "high",
      "note_count": 12,
      "avg_likes": 5200,
      "median_likes": 1800,
      "burst_count": 4,
      "evidence_note_ids": ["note_1", "note_2"]
    }
  ]
}
```

### 5.7 验收标准

- “其他/未分类”不再作为推荐主题。
- 如果未分类比例 > 50%，报告必须提示分类不足。
- 每个主题都能追溯到 note_ids。
- 主题表现统计来自真实样本。

## 6. 阶段三：新增 Insight Brief 中间层

### 6.1 目标

让报告、AI、前端都从同一份结构化洞察出发，避免散乱模板直接写报告。

### 6.2 新增模块

建议新增：

```text
backend/services/insight_brief.py
```

或先新增脚本：

```text
generate_insight_brief.py
```

### 6.3 输入

- `all_notes.json`
- `contents.json`
- `profile.json`
- `results.json`
- `data_quality.json`

### 6.4 输出

```text
data/tasks/{task_id}/insight_brief.json
```

结构：

```json
{
  "account_summary": {
    "nickname": "",
    "positioning": "",
    "one_sentence": "",
    "learnability": "medium",
    "data_confidence": "medium"
  },
  "data_quality": {},
  "strong_insights": [],
  "weak_insights": [],
  "topic_clusters": [],
  "title_patterns": [],
  "evidence_notes": [],
  "recommended_actions": [],
  "avoid_actions": [],
  "insufficient_modules": []
}
```

### 6.5 洞察对象结构

```json
{
  "id": "insight_1",
  "title": "老手机摄影实验是该账号的核心爆款主线",
  "summary": "高赞笔记集中在老设备、时间锚、画质反差这类内容上。",
  "confidence": "high",
  "sample_size": 7,
  "metric": {
    "avg_likes": 7196.3,
    "vs_overall_pct": 652.4
  },
  "evidence_note_ids": ["note_1", "note_2"],
  "why_it_matters": "用户被低成本旧设备带来的反差感吸引。",
  "action": "测试'旧工具 + 意外效果'选题。"
}
```

### 6.6 准入规则

强洞察：

- `sample_size >= 8`
- `confidence != low`
- 均值和中位数相对基准方向一致
- 单篇最高表现样本的组内互动贡献不超过 60%
- 至少包含支持样本、普通样本和反例
- `evidence_note_ids` 非空
- 不是“其他/未分类”
- 指标差异明显，不能仅由一篇爆款触发

弱洞察：

- 样本数为 3-7。
- 均值和中位数方向不一致。
- 单篇极端值贡献超过 60%。
- 差异不显著或证据只有单一表现层级。
- 语义判断依赖正文但正文不完整。

不输出：

- 没证据。
- 数据字段为空。
- 与指标相反。

### 6.7 验收标准

- 每条洞察都有 evidence_note_ids。
- 每条强洞察同时有 supporting、typical、counterexample 三类证据。
- 每条行动建议都能追溯到洞察。
- 没有证据的建议不会进入主报告。
- 小样本和单篇爆款驱动的模式只进入观察区。
- Markdown 和前端页面都能读取同一份 brief。

## 7. 阶段四：报告质量检查器

### 7.1 目标

自动阻止低质量报告交付。

### 7.2 新增模块

```text
backend/services/report_quality.py
```

或脚本：

```text
check_report_quality.py
```

### 7.3 检查项

数据检查：

- `content_span_months` 是否为负。
- `p25 <= median <= p75`。
- 点赞字段完整度是否过低。
- 标签为空时 SEO 是否降级。

结论检查：

- 是否把低表现模式作为推荐。
- 是否把“其他/未分类”作为最佳主题。
- 是否存在没有 evidence_id 的行动建议。
- 是否引用不存在的 note_id。
- 是否把低置信观察写成强结论。
- 是否只看均值而忽略中位数和离群值贡献。
- 是否出现数量与描述矛盾，例如“41 篇但产量低”。
- 是否在双方数值相同或均为 0 时生成方向性解释。
- 是否把局部正文/标签样本写成全量样本。
- 是否在没有搜索量数据时判断标签搜索量。
- 是否存在只有正例、没有普通样本和反例的强结论。

AI 检查：

- AI 是否引用不存在的标题。
- AI 是否新增了 brief 中没有的数据。
- AI 是否给出保证涨粉/保证爆款等承诺。

### 7.4 输出

```text
data/tasks/{task_id}/report_quality_check.json
```

结构：

```json
{
  "passed": true,
  "score": 86,
  "blocking_issues": [],
  "warnings": [
    "标签数据不足，SEO 模块已降级"
  ]
}
```

质量分拆为：

```json
{
  "data_completeness_score": 80,
  "evidence_sufficiency_score": 70,
  "conclusion_credibility_score": 65,
  "actionability_score": 75,
  "overall_score": 73
}
```

总分不能只由文件可读性和字段完整性计算。证据充分度或结论可信度低于 60 分时，不显示绿色“报告质量通过”。

### 7.5 验收标准

- 存在阻断问题时，报告页显示“本次报告数据异常”，并标出原因。
- 非阻断问题显示为数据提示。
- 质量检查结果能在前端展示。
- 单篇极端值驱动的强结论能被降级。
- 明显矛盾文案能阻断报告通过。

## 9. 阶段六：正文与标签采集增强

### 9.1 目标

优先补齐正文和标签，让 SEO、正文结构、内容表达分析不再长期降级。

不优先做封面图、图片理解或视频理解。原因是正文和标签的用户价值更直接，成本和风险更低。

### 9.2 当前问题

当前列表采集能稳定拿到：

- 标题。
- note_id。
- 笔记类型。
- 点赞数。
- 发布时间。
- 账号基础信息。

但正文和标签存在问题：

- 主页列表视图通常拿不到完整标签。
- 主页列表视图通常拿不到完整正文。
- 当前 `contents.json` 中的 `content_text` 存在点赞数字占位问题，不能当成正文使用。
- 因此 `content_text_completeness` 和 `tag_completeness` 必须以真实字段重新计算。

### 9.3 实现策略

采用“低频详情补采”，不做默认全量深度抓取。

第一版补采范围：

```text
Top 高赞样本 10 条
关键主题代表样本 10 条
低表现避坑样本 5 条
合计上限 20-30 条
```

触发方式：

- 默认分析仍先跑轻量主页报告。
- 报告页展示“补充正文与标签分析”按钮。
- 用户点击后才执行详情补采。
- 补采失败时报告保持可用，只显示数据不足。

### 9.4 重点文件

- `backend/services/collector.py`
- `backend/tasks/worker.py`
- `analyze_all.py`
- `backend/services/insight_brief.py`
- `backend/services/report_quality.py`
- `frontend/src/components/ProductReport.vue`
- `frontend/src/api/client.ts`

### 9.5 技术难点

1. 详情接口参数。
   - 单篇详情可能需要完整 note URL、`note_id`、`xsec_token`、来源上下文。
   - 列表数据必须保存足够字段，方便后续详情补采。

2. 速度。
   - 详情请求是一篇一篇补。
   - 20-30 条可接受，100 条默认补采会明显变慢。

3. 失败率。
   - Cookie 过期、字段变化、笔记不可见都会失败。
   - 需要按条记录失败原因，而不是让整个任务失败。

4. 风险控制。
   - 不做并发轰炸。
   - 控制补采数量。
   - 增加间隔和重试上限。
   - 不做批量账号监控。

5. 数据真实性。
   - 正文必须是真实正文。
   - 标签必须是真实标签。
   - 空字段不能被当成 0 或普通文本。

### 9.6 新增输出

```text
data/tasks/{task_id}/detail_fetch.json
data/tasks/{task_id}/contents.json
```

`detail_fetch.json` 示例：

```json
{
  "enabled": true,
  "requested_count": 25,
  "success_count": 18,
  "failed_count": 7,
  "failures": [
    {"note_id": "xxx", "reason": "detail unavailable"}
  ]
}
```

`contents.json` 要求：

```json
{
  "note_id": "",
  "content_text": "真实正文",
  "tags": ["真实标签"],
  "comment_count": 0,
  "collected_count": 0,
  "share_count": 0

```

### 9.7 验收标准

- `contents.json` 不再出现点赞数伪装正文。
- `content_text_completeness` 只统计真实正文。
- `tag_completeness` 只统计真实标签。
- 标签完整度 >= 50% 时，SEO 模块输出策略。
- 标签完整度 < 50% 时，SEO 模块继续降级。
- 正文完整度 >= 50% 时，正文结构模块输出模板。
- 正文完整度 < 50% 时，正文结构继续降级。
- 补采失败不会导致基础报告失败。
- 页面能显示补采进度和失败原因。

## 10. 阶段七：非 AI 专家分析层

### 10.1 目标

在不配置 API Key、不调用生成式 AI 的情况下，把当前报告从“统计结果 + 固定模板”升级为稳定、透明、可验证的规则型运营顾问。

阶段七优先解决的不是文风，而是判断质量：

- 能区分稳定规律、趋势信号、偶然爆款和数据不足。
- 能说明结论由哪些指标、正例、普通样本和反例共同支持。
- 能识别标题和正文中的结构特征，但不越界推断用户心理。
- 能把不确定结论改写成小规模验证实验。
- 能阻止数字与文案矛盾、样本口径错误和伪因果进入报告。

### 10.2 当前质量基线

以 `task_3` 的“电波曲奇”报告为回归基线，当前已确认的问题包括：

- 账号定位出现同义反复，不能回答具体内容价值。
- 41 篇主题内容被描述为“产量低、小而美”。
- “数字承诺结构”只有 3 条，其中一条贡献绝大多数点赞，却被写成显著有效。
- Top/Bottom 正文长度接近时仍生成“深度内容更易引爆”。
- 两组日期比例均为 0 时仍生成“时效性内容更容易推荐”。
- 标签只补采约 25 条，但报告按约 99 条样本表述并输出固定标签策略。
- 报告质量分 95 只代表技术检查通过，不能代表结论可信。

阶段七完成后，上述问题必须全部进入自动回归用例。

### 10.3 新处理链路

```text
results.json / contents.json
  ↓
文本结构特征与稳健统计特征
  ↓
insight_candidates 候选洞察
  ↓
结论准入与降级规则
  ↓
supporting / typical / counterexample 证据组合
  ↓
claim_registry 可信洞察
  ↓
insight_brief.json
  ↓
规则化人话表达 + 语义质量检查
```

Markdown 和前端不得再绕过 `claim_registry` 直接根据单个指标生成策略文案。

### 10.4 洞察候选结构

`insight_candidates` 至少包含：

```json
{
  "id": "candidate_title_1",
  "module": "title",
  "claim": "数字承诺结构可能存在表现信号",
  "facts": {
    "sample_size": 3,
    "avg_likes": 2576.3,
    "median_likes": 181,
    "baseline_median": 267,
    "top_note_share": 0.958
  },
  "level": "observation",
  "supporting_note_ids": [],
  "typical_note_ids": [],
  "counterexample_note_ids": [],
  "limits": [],
  "rule_trace": []
}
```

`rule_trace` 用于记录该结论为什么被提升、降级或拒绝，方便质量检查和后续调参。

### 10.5 P0：稳健结论引擎

重点文件：

- `analyze_all.py`
- `backend/services/insight_brief.py`
- `backend/services/report_quality.py`
- `generate_reports.py`

任务：

1. 为主题、标题公式和正文结构增加：
   - `median_likes`
   - `top_note_share`
   - `above_baseline_rate`
   - `sample_size`
   - `outlier_driven`

2. 实现三级结论：
   - `strong`
   - `observation`
   - `insufficient`

3. 强结论初始门槛：
   - 样本数 >= 8。
   - 均值和中位数方向一致。
   - 单篇最高表现样本贡献不超过 60%。
   - 至少存在支持样本、普通样本和反例。

4. 样本数 3-7、均值/中位数冲突或离群值驱动时，自动降级为“待测试观察”。

5. 一篇爆款不能单独触发“最佳主题”“显著有效”或“建议扩大占比”。

### 10.6 P0：分层证据选择器

为每条重要洞察选择：

- `supporting_note_ids`：支持结论的高表现样本，默认 2 条。
- `typical_note_ids`：同类普通表现样本，默认 2 条。
- `counterexample_note_ids`：同特征但不支持结论的反例，默认 1 条。
- `baseline`：账号整体中位数和同类基准。

证据评分考虑：

- 与当前结论的特征匹配度。
- 在同类中的表现层级。
- 是否已重复引用。
- 是否有正文、标签等完整字段。
- 是否能补充正例之外的信息。

默认证据区可以继续展示高赞样本，但强结论的证据组合不能只由高赞样本构成。

### 10.7 P1：规则化文本特征引擎

新增标题特征：

- 提问、断言、否定、对比、数字、时间锚。
- 人物、品牌、产品、场景和结果承诺。
- 第一/第二人称、引用、情绪词和冲突词。
- 信息型、观点型、经历型、教程型、测评型表达。

新增正文特征：

- 开头类型：提问、故事、结论、数据、背景。
- 段落数、句长、正文长度、列表结构。
- 步骤、案例、引用、对比、转折、总结。
- 行动提示、评论引导、关注引导。
- 标题承诺是否在正文中得到回应。

实现原则：

- 使用分词、正则、关键词词典和段落结构完成第一版。
- 只输出可观察结构，不推断受众心理和平台推荐原因。
- 正文覆盖不足时按真实匹配样本数降级，而不是只看全局覆盖率。

### 10.8 P1：账号专属主题聚类

在现有宽关键词分类基础上增加非 LLM 聚类：

1. 从标题、标签和正文提取高频词与共现词。
2. 使用 TF-IDF、字符 n-gram 或相似度矩阵形成候选簇。
3. 用对象词 + 场景词 + 内容意图组合主题名。
4. 保存主题关键词、代表标题和命名来源。
5. 无法稳定归类的样本继续保留为未归类。

第一版不追求分类覆盖率最大化，优先保证主题名称可解释、样本归属可复查。

### 10.9 P1：实验型行动建议

`recommended_actions` 增加：

```json
{
  "source_insight_id": "insight_x",
  "variable": "标题是否使用明确结果承诺",
  "test_count": 3,
  "control": "选题和内容形式尽量接近",
  "metrics": ["24小时互动", "中位数", "评论反馈"],
  "scale_condition": "至少 2 篇高于当前中位数",
  "stop_condition": "连续 3 篇低于当前中位数"
}
```

强结论可以给出“优先测试”；趋势观察只能给出“小规模验证”；数据不足只提示补数据或暂不判断。

### 10.10 P0：语义质量检查

`report_quality_check.json` 增加四个分项：

- 数据完整度。
- 证据充分度。
- 结论可信度。
- 行动可执行度。

新增阻断规则：

- 数量与描述矛盾。
- 双方数值相同却生成高低判断。
- 单篇极端值驱动却输出强结论。
- 局部补采样本被写成全量。
- 没有平台搜索数据却判断搜索量。
- 强结论缺少普通样本或反例。
- 行动建议没有测试数量、指标或停止条件。

证据充分度或结论可信度低于 60 分时，前端不得显示绿色“报告质量通过”。

### 10.11 阶段七验收标准

- “电波曲奇”不再输出“41 篇但产量低”。
- 3 条且由单篇爆款驱动的标题模式被标记为待测试观察。
- Top/Bottom 指标相同或均为 0 时不生成方向性原因。
- 标签模块显示真实详情样本数，25 条不能写成 99 条完整标签样本。
- 每条强洞察至少包含 2 条支持样本、1 条普通样本和 1 条反例。
- 每条建议包含测试变量、测试数量、观察指标和停止条件。
- 报告明确区分事实、观察、假设和验证动作。
- 质量分能够因语义矛盾、离群值驱动和证据不足而下降。
- 未配置 API Key 时，规则版报告仍可独立完成上述交付。

## 11. 阶段八：可选 AI 解读

### 11.1 目标

在不牺牲稳定性的前提下，让报告更像“懂小红书的人在给新手解释”。

### 11.2 AI 输入

只允许输入：

- `insight_brief.json`
- 必要的证据样本标题和摘要。
- 数据质量说明。

不允许 AI 直接从散乱原始数据自由发挥。

### 11.3 AI 输出

新增：

```text
data/tasks/{task_id}/ai_summary.json
```

结构：

```json
{
  "plain_language_summary": "",
  "what_to_learn": [],
  "what_not_to_copy": [],
  "seven_day_plan": [],
  "confidence_notes": []
}
```

### 11.4 提示词原则

- 只基于给定 JSON。
- 不新增数据。
- 不引用不存在的标题。
- 面向新手。
- 每条建议绑定 evidence_id。
- 数据不足必须明说。

### 11.5 降级策略

- 未配置 LLM：不显示 AI 解读。
- LLM 调用失败：显示规则版报告。
- AI 输出校验失败：丢弃 AI 输出，显示规则版报告。

### 11.6 验收标准

- AI 不是必需依赖。
- AI 失败不会影响任务完成。
- AI 生成内容更自然，但不越过数据事实。

### 11.7 当前规则版为什么不依赖 AI

当前主报告虽然是自然语言，但不是 LLM 生成。

生成机制：

```text
results.json 计算事实
  ↓
insight_brief.json 组织洞察
  ↓
ProductReport.vue 按用户阅读路径展示
```

自然感来自：

- 主题、标题、爆款率等结构化指标。
- 人工写好的洞察字段模板。
- 前端按“判断 -> 证据 -> 行动”的顺序组织。
- 质量检查器阻止不可靠结论进入主报告。

AI 后续只作为表达增强，不作为事实来源。

- 行动建议包含测试数量、观察指标和停止条件。
- 质量分能反映证据充分度和结论可信度。
- AI 可选，不影响基础报告可用性。
- 报告能稳定服务“新手博主下一步怎么做”。

## 16. 阶段十：动态主题发现层

### 16.1 目标

把主题识别从“固定词典匹配为主”升级为“账号内动态主题发现为主”。固定规则只做辅助命名、质量校验和兜底，避免每遇到一个新垂类就继续补词典。

### 16.2 当前问题

现有规则层存在结构性风险：

- 未进入词典的新主题会被放进“未归类观察”。
- 弱关键词会误伤，例如“抄作业版”触发教育类。
- 报告首页强结论依赖主题分类，一旦主题错，后续洞察和行动建议都会被带偏。
- 继续补关键词会变成无穷无尽的维护工作。

### 16.3 技术方案

第一版不引入 embedding 和外部 AI，采用本地轻量算法：

```text
analysis_text = 标题 + 已采集正文 + 标签
  -> 中文短语 / 字符 n-gram 抽取
  -> 停用词与弱片段过滤
  -> 词频、共现、互动表现加权
  -> 贪心聚类形成候选主题
  -> 规则词典辅助命名
  -> 主题置信度评分
  -> 低置信主题降级
```

### 16.4 重点文件

- `analyze_all.py`
- `backend/services/insight_brief.py`
- `backend/services/report_quality.py`
- `tests/regression_topic_classification.py`

### 16.5 实现任务

1. 在 `analyze_all.py` 中新增动态主题发现：
   - 从每条笔记生成 `_topic_terms`。
   - 按账号内词频、共现和点赞表现计算候选词分数。
   - 使用高分词作为 seed 聚类相似笔记。
   - 对无法稳定命名但样本足够的主题，命名为“围绕「关键词」的内容线”。

2. 固定规则降级：
   - 规则不再作为唯一分类来源。
   - 规则用于辅助动态主题命名。
   - 剩余未聚类样本才进入规则兜底。

3. 输出结构补强：
   - `topic_source`
   - `matched_keywords`
   - `confidence`
   - `evidence_note_ids`
   - `evidence_sort_rule`

4. 置信度门禁：
   - 样本数不足的动态主题不进入主题地图。
   - 低信息主题不进入首页强结论。
   - 非规则命名动态主题默认至少需要 5 条样本。

5. 回归测试：
   - 老手机影像必须形成高置信主题。
   - “抄作业版”不能误判为教育学习。
   - 固定词典未覆盖的新领域，例如陶瓷杯/釉色，样本足够时必须形成动态主题。

### 16.6 验收标准

- `python tests/regression_topic_classification.py` 通过。
- `task_5` 中“手机摄影技巧与审美”恢复为高置信主题。
- `task_5` 未归类比例低于 50%。
- 报告首页不再显示“本次数据暂未形成稳定高表现主题”这类错误降级。
- 主题地图不展示由明显碎片词拼出的低样本动态主题。
- 新垂类不需要先手动补词，也可以在样本足够时形成可解释主题。

### 16.7 后续增强

第二版可以考虑：

- 引入 `jieba` 或更稳定的中文分词。
- 使用 TF-IDF 替代当前轻量 n-gram 分数。
- 加入标题、正文、标签不同权重。
- 在配置 AI 后，仅让 AI 对动态主题做命名润色，不允许 AI 新增主题事实。

## 18. 阶段十二：移除用户侧 Node.js 依赖

### 18.1 目标

把前端从开发服务器模式改为静态资源模式，先去掉用户必须安装 Node.js 才能打开网页这一层门槛。

当前形态：

```text
start_windows.bat
  -> 启动 backend: 127.0.0.1:8000
  -> 启动 frontend vite dev server: 127.0.0.1:5173
  -> 插件打开 5173
```

目标形态：

```text
start_windows.bat
  -> 启动 backend: 127.0.0.1:8000
  -> backend 直接托管 frontend/dist
  -> 插件打开 8000
```

### 18.2 实现任务

优先级：P0

1. 前端构建产物接入后端：
   - 执行 `npm run build` 生成 `frontend/dist`。
   - FastAPI 增加静态文件托管。
   - `/` 返回前端首页。
   - `/assets/*` 返回前端资源。
   - API 路由保持 `/api/*`。

2. 修改前端 API base URL：
   - 生产模式下使用同源 `/api`。
   - 开发模式下仍可使用 `127.0.0.1:8000`。

3. 修改插件地址：
   - `FRONTEND_BASE` 从 `http://127.0.0.1:5173` 改为 `http://127.0.0.1:8000`。
   - `host_permissions` 保留 `8000`，`5173` 仅开发用。

4. 修改启动器：
   - 不再启动 `npm run dev`。
   - 如果存在 `frontend/dist`，直接启动后端。
   - 开发环境可保留一个 `start_dev_windows.bat` 或开发说明。

5. 修改打包脚本：
   - 打包前自动执行 `npm run build`。
   - zip 中包含 `frontend/dist`。
   - zip 中不需要 `frontend/node_modules`。

### 18.3 关键风险

- 前端路由刷新可能 404，需要 FastAPI 对非 API 路由回退到 `index.html`。
- 静态资源路径需要兼容 Windows 路径和 PyInstaller 解包路径。
- 插件和前端地址变更后，旧版插件可能仍打开 `5173`，需要同步版本号和说明。

### 18.4 验收标准

- 用户电脑不安装 Node.js，也可以启动本地网页。
- `start_windows.bat` 只启动一个本地后端进程。
- 打开 `http://127.0.0.1:8000` 能看到前端首页。
- 插件点击“打开报告”进入 `8000` 页面。
- `npm run build` 仍在开发机和打包机执行，而不是在用户电脑上执行。

## 19. 阶段十三：Python 后端打包为 Windows exe

### 19.1 目标

让用户不再需要安装 Python。交付物从“源码 + Python 环境”变成“本地可执行程序 + 插件”。

目标形态：

```text
xhs-blogger-analyzer-windows/
  XHS分析助手.exe
  browser-extension/
  docs/
  data/
```

用户只需要：

```text
解压 -> 双击 XHS分析助手.exe -> 浏览器自动打开 -> 安装插件 -> 分析
```

### 19.2 实现任务

优先级：P0

1. 选择打包方案：
   - 优先评估 PyInstaller。
   - 备选 Nuitka。
   - 先做目录模式 `--onedir`，稳定后再考虑 `--onefile`。

2. 整理运行时路径：
   - 数据目录：`data/`。
   - 日志目录：`logs/`。
   - 静态资源：`frontend/dist`、`static`、`spider_xhs/static`。
   - 配置文件：`.env.example` 或默认配置。

3. 修改后端路径读取：
   - 支持源码运行路径。
   - 支持 PyInstaller `_MEIPASS` 或 exe 旁边资源路径。
   - 所有写入文件都写到 exe 同级目录下的 `data/` 和 `logs/`，不写进程序内部。

4. 处理采集签名依赖：
   - 当前采集链路可能依赖 JS 执行环境。
   - 如果 `PyExecJS` 依赖用户电脑 Node.js，则必须解决：
     - 方案 A：改用 Python 内可打包的 JS runtime，例如 quickjs。
     - 方案 B：随包携带 portable Node.js。
     - 方案 C：把签名逻辑迁移为纯 Python 或可嵌入运行时。
   - 阶段十三不能只把 Python 打成 exe 后仍要求用户安装 Node.js，否则低门槛目标没有达成。

5. 增加 exe 启动逻辑：
   - 启动 FastAPI。
   - 自动打开浏览器。
   - 如果端口占用，提示原因。
   - 退出方式明确。

6. 打包脚本：
   - 新增 `scripts/build_windows_exe.ps1`。
   - 自动执行前端 build。
   - 自动执行 PyInstaller。
   - 输出 `release/xhs-blogger-analyzer-windows/`。

### 19.3 关键风险

- PyInstaller 可能漏收动态导入模块。
- `spider_xhs` 下的 JS 静态文件体积较大，打包产物会变大。
- 杀毒软件可能误报本地服务 exe。
- 没有代码签名时，Windows 可能提示未知发布者。
- 采集签名依赖 JS runtime 是最大不确定性，必须优先验证。

### 19.4 验收标准

- 干净 Windows 测试机不安装 Python，也能运行 exe。
- 不安装 Node.js，也能完成一次真实分析。
- exe 自动打开本地页面。
- 插件能连接本地服务并创建分析任务。
- 日志、数据库、报告都写入 exe 同级 `data/` 或 `logs/`。
- 删除整个文件夹即可卸载，不残留系统服务。

## 22. 交付路线总览

### 22.1 推荐顺序

```text
阶段十一：一键启动脚本固化
阶段十二：前端静态化，移除用户侧 Node.js 依赖
阶段十三：Python 后端打包 exe，移除用户侧 Python 依赖
阶段十四：Windows 桌面内测包
阶段十五：安装器 + 插件商店 + 正式分发
```

### 22.2 当前最优先的下一步

不要直接做安装器。下一步优先做阶段十二：

```text
前端 build 成静态文件
后端托管前端
本地只启动一个端口
用户不再需要 Node.js
```

原因：

- 这是最低成本、最高收益的一步。
- 能明显降低用户环境门槛。
- 也为后续 PyInstaller 打包 exe 铺路。
- 如果不先做静态化，exe 打包后仍要启动 Vite dev server，交付形态会很别扭。

### 22.3 关键决策点

阶段十三开始前必须确认：

```text
采集链路是否仍依赖外部 Node.js 执行 JS
```

如果依赖，则必须先选择：

- 改用可嵌入 JS runtime；
- 或随包携带 portable Node.js；
- 或重写签名执行层。

这个决策会直接决定“是否真的能做到用户无需安装 Node.js”。
